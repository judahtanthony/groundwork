Local-first agent control plane · human web UI

A focused developer tool for planning and supervising AI coding agents.

A human plans and reviews work as a DAG of work-nodes; agents execute inside approved envelopes. This system defines the visual language, component library, and the two surfaces — the Roots board and the up·here·down node view — plus the load-bearing interactions and every state. Built to ship as a static SPA embedded in one binary.

Static Vite SPA · go:embed Request/response + light polling Server is source of truth CLI parity Light + dark · WCAG-AA

06 — State coverage

Every list, panel, and action, in all six states

Request/response with light polling means loading and error are first-class, not afterthoughts. Blocked and waiting are deliberately two different colors; ineligible actions render disabled with the reason inline, never hidden.

Empty

Create your first root
Roots are parentless initiatives, typically human-created.

Loading

skeleton

Panes keep last-known value — never blank-flash populated content.

Blocked / ineligible

Blocked — input requested

Durable decision record: "Which migration strategy — expand-contract or dual-write?" Answering makes the node eligible and resumes a new run.

Error

not_a_repo
This path isn't a git repository. Land preview can't run.

The contract {error:{code,message}} renders inline & actionable — never a raw stack.

Pending-approval

Approval requestedinbox

The same gate appears three ways — a badge on the node, a count in the header strip, and an inbox entry — all deep-linkable.

Active-run

Active runimplementationplanning

Indicated on the node, in the children column, and in the board row's attention badge. Planning vs implementation runs are badged differently; opens run detail.

Responsive — narrow window

Below ~900px the three-zone spine collapses to a single focused column: ancestors become a back-chip row, children move to a bottom sheet reachable by a "Down · N children" bar, and the action rail docks to the footer. The Roots board keeps its rows, dropping the owner glyph and progress bar first, then folding attention pills into a single count. No separate mobile IA — the same routes, reflowed.

07 — Handoff

Exportable design tokens

Everything here is driven by one token set with a light and a dark map. Consume the CSS custom properties directly, or the JSON for a build step. Both live in /tokens.

Ships as

Static Vite build + go:embed. No SSR, no runtime Node. Theme via [data-theme] on the root, persisted to localStorage.

Realtime-ready

Request/response + light polling now. The digest, active-run, and pending states are polled; the same components accept a WebSocket push later with no rewrite.

CLI parity

Every action maps to a gw command; the UI is a client of the JSON API and never bypasses a gate. export/import/sync stay CLI-only in v1.
